## 專案除錯經理報告
### 目前確認的主要問題
- 教學模式仍殘留部分舊的遭遇卡 / AR 流程，導致玩家在主線中看到不該出現的白底互動卡。
- 教學模式原本採「直接自動放行」，LM 雖然已接上專案，但在教學主線中幾乎沒有真正參與內容回覆，玩家只感覺到流程跳關。
- `ai-lab` 部分 API 在網路中斷或本機服務未啟動時，直接把 `Failed to fetch` 暴露給玩家，缺乏 RPG 語境與可理解的錯誤演出。
- NPC 對話雖然已有名字，但角色辨識度仍不足，玩家不容易感受到不同角色真的在出場。
- service worker 與前端快取版本若未同步升級，手機可能持續載入舊版 `ai-lab.js`，導致「明明修了但畫面沒變」。

### 建議移除 / 收斂的內容
- 故事模式中的 `taskEncounterModal / taskTargetObj / taskGuideArrow` 只應保留給大富翁或特殊探索格，不應進入教學主線。
- 教學模式中的「純自動放行」邏輯應改為「結果強制放行，但 LM 仍回覆內容」。
- 所有直接把技術錯誤字串顯示給玩家的路徑應收斂為探索艙 / NPC 語境。

## 技術經理執行清單
### 第一層：流程穩定性
- 封鎖教學 / 劇情主線中的遭遇卡殘留。
- 補強故事模式切關、切任務、切快取時的 UI 收斂。
- 補 service worker / 前端資源版本，避免手機吃舊檔。

### 第二層：LM 真正參與
- 教學模式的 AI 圖片任務改為「LM 先看圖回覆，再強制通關」。
- 前端優先展示 LM 的 `reason`，而不是只顯示通關訊息。
- 若 LM 暫時失敗，仍保留教學流程，但要有明確 fallback 文案。

### 第三層：玩家可理解錯誤
- 把 `Failed to fetch` 之類的錯誤改寫成玩家看得懂的探索艙提示。
- API 送出失敗時以 NPC 救援員文案接手，而不是只吐技術錯誤。

### 第四層：RPG 角色演出
- 加強 NPC 對話框角色辨識度。
- 讓關主 / 裁判 / 主持人 / 救援員 / 導覽員有更明顯的視覺差異。
